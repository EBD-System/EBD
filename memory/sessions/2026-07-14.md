# Sessão 2026-07-14

## O que foi alterado

- A base do projeto foi consolidada em PostgreSQL.
- A configuração do frontend passou a apontar para uma URL genérica de API HTTP que deve ser conectada ao banco.
- As mensagens de erro e os comentários do cliente foram atualizados para falar em backend PostgreSQL.
- A memória consolidada foi ajustada para registrar o PostgreSQL como fonte oficial.

## Conhecimento consolidado

- O projeto não deve carregar dependência nominal de base antiga na camada de interface.
- Ainda falta a ponte HTTP entre o frontend e o PostgreSQL para o site funcionar de ponta a ponta.

## Próximos passos

- Conectar a interface a uma API real que leia/escreva no PostgreSQL.
- Validar as rotas e os fluxos de presença depois que o endpoint estiver disponível.

## Atualização 2026-07-14 (separação do fluxo de login e sessão)
### O que foi alterado
A navegação passou a tratar a autenticação como sessão persistida no navegador, com identidade, perfis e modo de acesso derivados do estado salvo. O parâmetro `?code=` permanece apenas como ponte de migração, e as páginas dedicadas de aluno deixaram de depender dele como regra principal.

### Conhecimento consolidado
A sessão é a fonte principal de acesso da interface. O login inicial pode vir de um identificador legado, mas a navegação interna deve ler o estado autenticado salvo, não a URL. A identidade de usuário e os perfis precisam ser preservados na sessão para apoiar autorização no cliente e no backend.

### Próximos passos
Quando o endpoint de autenticação estiver disponível, substituir o login legado por retorno de sessão com `userId`, `login`, `nome`, `perfis` e `accessMode`, mantendo o guard de rotas no cliente como primeira barreira.

## Atualização 2026-07-14 (correção de navegação no GitHub Pages)
### O que foi alterado
O roteador e os links internos passaram a preservar o basename `/EBD` do deploy no GitHub Pages. Também foi criado um bypass temporário centralizado (`DEV_BYPASS_AUTH`) para testar as rotas internas sem login.

### Conhecimento consolidado
Qualquer navegação interna precisa passar por um helper que reconstrói a URL com o basename correto. Rotas e redirects não podem usar `/login` ou outras URLs absolutas como se a aplicação estivesse na raiz do domínio.

### Próximos passos
Quando a autenticação real ficar pronta, desligar `DEV_BYPASS_AUTH` e validar novamente todas as rotas no deploy publicado.
