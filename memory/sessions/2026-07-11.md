# Sessão 2026-07-11

## O que foi alterado
O fluxo de edição de aluno foi endurecido em quatro pontos: o carregamento da aplicação passou a usar `apiGet` com timeout; o envio de atualização de aluno ganhou fallback automático para GET quando o POST responde `Ação inválida`; o cliente passou a enviar `acao` como alias de compatibilidade; e o backend agora aceita atualização por chave derivada do nome, preservando turma/status atuais quando o payload vem incompleto.

## Conhecimento consolidado
No Apps Script, a atualização de aluno deve aceitar a chave derivada do cadastro e não depender de um `idaluno` separado na planilha. Para reduzir falhas de roteamento, o cliente precisa tolerar tanto `doPost` quanto `doGet` no mesmo endpoint, e o carregamento inicial não deve ficar preso em fetch sem timeout.

## Próximos passos
Se a edição continuar falhando, verificar o deploy ativo do Apps Script e confirmar se a URL publicada aponta para a versão atual do backend.

## Atualização 2026-07-11 (cadastro de aluno)
### O que foi alterado
O backend recebeu reforço no fluxo de inclusão de aluno: a ação `addAluno` continua gravando na aba `Cadastro`, ganhou aliases adicionais de rota e passou a expor um helper manual `adicionarAlunoManual()` para testes no editor do Apps Script. O frontend já estava alinhado com a ação `addAluno` e segue usando o formulário de novo aluno na página principal.

### Conhecimento consolidado
O cadastro rápido de aluno é um fluxo separado da edição: a UI envia `addAluno`, o backend persiste em `Cadastro` e o modo manual de validação pode chamar o helper direto no Apps Script.

### Próximos passos

### Ajuste adicional
O backend passou a localizar a aba de cadastro sem depender de maiúsculas/minúsculas no nome da guia, evitando duplicação entre `Cadastro` e `cadastro`.


## Atualização 2026-07-11 (fallback de fetch)
### O que foi alterado
O cliente passou a repetir como `GET` as ações `addAluno`, `addTurma` e `updateAluno` quando o `POST` falha com `Failed to fetch` ou com erro de ação inválida. Isso reduz falhas de cadastro no Web App do Apps Script quando o POST não atravessa corretamente o `fetch` do navegador.

### Conhecimento consolidado
No deploy atual, o endpoint do Apps Script deve continuar aceitando a mesma rota em `doGet` e `doPost`, porque o front pode precisar cair para `GET` em ações de escrita.

### Próximos passos
Se o erro persistir, validar a URL publicada em `APPS_SCRIPT_URL` e conferir se o deploy ativo ainda é o mais recente.

## Atualização 2026-07-11 (adicionar aluno em página dedicada)
### O que foi alterado
Foi criada a rota `aluno/adicionar-aluno/` com formulário próprio para inclusão de aluno, sem a etapa de criar nova turma nessa tela. A tela principal ganhou um botão de atalho para essa rota, e o backend passou a aceitar `dataNascimento` opcional no cadastro de aluno.

### Conhecimento consolidado
O cadastro de aluno agora tem um fluxo dedicado para inclusão rápida, separado da edição. A data de nascimento é opcional e, quando informada, deve ser persistida junto com o mês de nascimento. A tela dedicada não deve reintroduzir cadastro de turma. O fluxo de inclusão também ficou aberto para qualquer modo de acesso.

### Próximos passos
Se o fluxo de inclusão precisar reaproveitar mais campos no futuro, expandir primeiro a página dedicada e só depois sincronizar o formulário rápido da tela principal.
